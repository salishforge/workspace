### Core Principle: Dynamic Loading First

**All configuration in Hyphae should:**

1. Load from files/sources at **request time** (not at startup)
2. Changes take effect **immediately** (no restart required)
3. Support full **rollback** (revert to previous state)
4. Maintain complete **audit trail** (track all changes)

**Why This Matters:**
- Services don't need restart for config changes
- Lower operational friction
- Better observability (can track config versions)
- Easier recovery from bad configurations
- Admin always in control

### Configuration Categories

**Dynamic Configurations** (Preferred)
- Agent override policies
- Model service endpoints
- Rate limits and quotas
- Feature flags
- Approved model lists
- Cost budget thresholds
- Any parameter that can change request-to-request

**Implementation:** Load from file on each request (e.g., `getPolicy()` reads `policies-config.json` every time)

**Result:** Changes active immediately, no restart needed

**Static Configurations** (Rare, Explicitly Flagged)
- Database connection strings
- HTTP server port numbers
- Process-level resource limits
- Encryption key material
- TLS certificate paths
- Inter-service communication protocols

**Implementation:** Read once at startup, stored in process memory

**Requirements:** Must be explicitly flagged in code, require admin confirmation before change, show affected services, recommend restart window

### Safe Configuration Change Workflow

**When admin changes a parameter:**

1. **Syntax & Structure Validation**
   - JSON parses correctly
   - Schema structure matches
   - Required fields present
   - Types are correct
   - Values within ranges

2. **Change Identification**
   - Which parameters changed?
   - Static or dynamic?
   - Major or minor change?

3. **Risk Assessment**
   - LOW RISK: Minor adjustments (<20%), dynamic params → Apply immediately
   - MEDIUM RISK: Major changes (>50%), list changes → Test in sandbox first
   - HIGH RISK: Static params, database changes → Require admin approval + testing + restart

4. **Testing** (for medium/high risk)
   - Spin up sandbox environment
   - Run test suite
   - Verify fallback behavior
   - Check performance
   - Measure latency

5. **Admin Approval** (for high risk)
   - Alert admin with detailed information
   - Show what changed
   - Show impact/affected services
   - Provide recommendation
   - Allow override "do it anyway"

6. **Apply to Production**
   - Create backup of current config
   - Write new configuration to file
   - Log change to audit trail
   - Alert admin with status
   - Provide rollback command

7. **Immediate Effect**
   - Dynamic: Changes active on next request
   - Static: Changes pending restart
   - Rollback: Always available via backup system

### Comparison: Old vs New Approach

**Old Approach (Startup-Loaded):**
- Config read at startup
- Changes require restart
- Admin downtime for updates
- Limited rollback options
- Lower observability

**New Approach (Dynamic Loading):**
- Config read on each request
- Changes immediate
- Zero planned downtime
- Easy rollback (revert file)
- Full audit trail
- Admin always in control

### Design Principles Established

1. **Dynamic Loading First** — Always prefer dynamic loading over startup loading
2. **No Restart For Config** — Users shouldn't need restart for config changes
3. **Safe Changes** — Validate, test, approve, apply workflow
4. **Full Audit Trail** — Every change logged with timestamp/actor/reason
5. **Easy Rollback** — Revert to any previous state instantly
6. **Admin Control** — Always require human approval for risky changes
7. **Clear Feedback** — Detailed alerts showing impact/recommendation
